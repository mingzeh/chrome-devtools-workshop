Chrome DevTools MCP Server 实战手记

MCP（Multi-Channel Protocol）让 DevTools 不再只是浏览器的附属面板，而是一个可编排的调试服务。通过 Server ，我们可以让脚本、IDE、Agent 共享同一个调试上下文，缩短排查链路。

查看工作流 与我讨论实现

Overview

MCP Server 为何重要

Connected

统一上下文

DevTools 与 CLI 共享同一份会话信息，调试命令可以直接读取浏览器状态，不需要重复采集。

Automation

自动执行策略

通过 MCP 将性能面板、Network、Console 的操作封装成脚本，结合 AI Agent 自动运行。

Collaboration

跨团队调试

后端、前端与测试可以同时接入 MCP Server，查看同一条 Trace 与诊断结论，实现“调试即文档”。

Workflow

我在 Workshop 中搭建的 MCP 工作流

01 · 连接通道

使用 Chrome DevTools 提供的 MCP Server 插件，先与浏览器建立安全通道，再通过 CLI 注册到同一个会话空间。

02 · 脚本化操作

把 Network 过滤、性能录制、DOM 快照等操作封装成脚本，命名清晰，方便 Agent 调用。

03 · Insights 输出

脚本抛出的结果通过 MCP 返回到 IDE，形成结构化报告，可直接贴到代码评审或缺陷跟踪系统。

一句话总结： DevTools MCP 让浏览器调试拥有了 API 化的接口——我可以把“经验动作”写成指令，再交给 Agent 或同伴重复执行。

Use Cases

我已经计划落地的场景

后端接口性能守护

通过 MCP 自动触发链路压测，同时抓取 Waterfall，与服务端日志比对，快速定位慢查询。

回归测试伴侣

在每次回归前执行固定的 DevTools 脚本，验证关键页面的 LCP、CLS 是否回退，异常直接同步到团队频道。

可观测性训练场

将前端事件与后端指标关联起来，在 MCP 中标记里程碑，帮助新人快速理解系统结构。

Agent Pairing

让 AI Agent 接手重复调试动作，人类聚焦在策略设计，确保效率与准确性兼得。
